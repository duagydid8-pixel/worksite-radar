# XERP 공수 가드레일 설계

## 목표
XERP 공수 반영에서 시간 입력이 공수로 바뀌는 과정에 실수가 들어가지 않도록 한다.

핵심 원칙은 세 가지다.

1. 공수는 가능한 한 자동 계산한다.
2. 계산 결과가 이상하면 저장 전에 막는다.
3. 사람이 건드릴 수 있는 값은 명시적 예외로만 허용한다.

## 문제 정의
현재 공수 반영 흐름은 업로드, 수동 수정, 다운로드, XERP&PMIS 저장이 한 화면에 함께 있다. 이 구조는 편하지만, 다음 실수에 취약하다.

1. 시간 문자열이 잘못 들어가도 공수로 조용히 변환되는 경우
2. 수동으로 고친 값이 저장/다운로드/반영 과정에서 다시 덮이거나 누락되는 경우
3. 경계 시간대나 예외 사유가 규칙과 다르게 처리되는 경우

이 설계의 목적은 “정확히 계산하는 기능”보다 “틀린 값이 최종 반영되지 못하게 막는 기능”을 만드는 것이다.

## 설계 방향

### 1. 단일 공수 판정 엔진
시간 계산, 가산 여부, 예외 사유 판정을 화면 코드 안에 흩어두지 않고 하나의 순수 함수로 모은다.

이 엔진은 다음을 반환한다.

1. 정규화된 출근/퇴근 시각
2. 계산된 공수값
3. 가산 필요 여부
4. 유효성 문제 목록
5. 저장 차단 여부

이 방식의 장점은 계산 규칙을 한 군데만 고치면 업로드, 저장, 다운로드, XERP&PMIS 반영이 모두 같은 결과를 쓰게 된다는 점이다.

### 2. 세 군데에서 검증
검증은 업로드 직후, 수동 편집 직후, 저장/반영 직전에 다시 수행한다.

업로드 직후에는 파싱 실패, 시간 형식 오류, 비정상 값 범위를 잡는다.

수동 편집 직후에는 사용자가 수정한 값이 기존 규칙과 충돌하는지 확인한다.

저장/반영 직전에는 마지막으로 전체 행을 재검사해서, 하나라도 차단 사유가 있으면 진행을 멈춘다.

### 3. 상태를 3단계로 분리
각 행은 아래 상태 중 하나를 가진다.

1. `valid`: 그대로 저장 가능
2. `warn`: 저장은 가능하지만 사람이 다시 봐야 함
3. `blocked`: 저장/반영 금지

경계 시간대, 예외 승인, 수동 보정처럼 해석이 필요한 항목은 `warn`으로 두고, 시간 형식 오류나 계산 불일치처럼 결과가 불확실한 항목은 `blocked`로 둔다.

### 4. 수동 수정은 명시적 예외로만
수동으로 `가산B`, `가산사유`를 수정하는 기능은 유지하되, 다음 조건을 붙인다.

1. 수정 전후 차이를 화면에서 보여준다.
2. 수동 수정은 자동 계산을 덮는 예외라는 점을 명시한다.
3. 저장 시에는 수동 수정 사유가 있어야 한다.
4. 예외가 붙은 행은 히스토리에 남긴다.

즉, 사용자가 값을 바꿀 수는 있지만 “조용히” 바뀌지 않도록 만든다.

## 동작 규칙

### 업로드
업로드된 각 행에 대해 시간 문자열을 정규화한다.

다음 항목은 즉시 차단한다.

1. 출근/퇴근 값이 시간으로 해석되지 않음
2. 공수 계산에 필요한 필드가 비어 있음
3. 같은 행 안에서 자동 계산 결과와 원본 값이 심하게 충돌함

다음 항목은 경고만 둔다.

1. 조출/연장 경계 시간
2. 예비군, 교육, 승인형 사유
3. 사람이 수동으로 보정했지만 허용 범위 안에 있음

### 저장
저장은 마지막 방어선이다.

저장 직전 전체 행을 다시 계산하고, blocked 행이 하나라도 있으면 저장을 취소한다. 이때 단순 실패 메시지 대신 차단 사유 수를 요약해서 보여준다.

예시 메시지:

`저장 불가: 시간 형식 오류 2건, 공수 불일치 1건`

### 다운로드
다운로드 파일은 화면의 현재 상태를 그대로 내보내되, blocked 행이 있으면 다운로드 자체를 막는다.

이유는 다운로드된 파일이 다시 반영될 가능성이 있기 때문이다. 틀린 파일을 밖으로 내보내는 것 자체를 금지한다.

### XERP&PMIS 반영
외부 반영은 가장 엄격하게 처리한다.

1. blocked 행이 있으면 반영 금지
2. warn 행은 반영 가능하지만 반영 직전에 확인 모달 표시
3. 반영 성공 후에는 반영 시각과 대상 건수를 히스토리에 저장

## 데이터 모델
새 공통 타입을 추가한다.

```ts
type ValidationSeverity = "warn" | "blocked";

interface GongsuValidationIssue {
  code: string;
  severity: ValidationSeverity;
  message: string;
}

interface GongsuValidationResult {
  isValid: boolean;
  isBlocked: boolean;
  issues: GongsuValidationIssue[];
  canonicalGongsu: number | null;
  canonicalReason: string;
}
```

행 데이터에는 최소한 다음 상태를 붙인다.

1. `validationIssues`
2. `validationState`
3. `canonicalGongsu`
4. `manualOverrideReason`

이 값들은 화면 표시용이 아니라 저장/반영 판단용으로도 쓴다.

## 테스트 전략
다음 케이스는 반드시 포함한다.

1. 07:00, 07:10, 07:11 경계 판정
2. 16:20, 17:00, 17:01 경계 판정
3. 공백, 잘못된 문자열, 숫자형 시리얼 값 파싱
4. 자동 계산값과 수동 수정값 충돌
5. blocked 행이 있을 때 저장/다운로드/반영 차단
6. warn 행은 허용되지만 확인이 필요한지 표시
7. 같은 입력이 업로드, 저장, 다운로드, 반영에서 동일한 결과를 내는지

## 비목표
이번 단계에서 하지 않을 일은 다음과 같다.

1. 공수 규칙 자체를 새로 정의하는 것
2. 전체 근태/급여 계산 엔진을 재작성하는 것
3. UI 전체를 다시 디자인하는 것

이번 작업은 규칙 변경이 아니라, 기존 규칙이 틀리지 않게 잠그는 작업이다.

## 완료 기준
아래 조건이 모두 만족되면 이 설계는 완료로 본다.

1. 시간 형식 오류가 저장/반영까지 도달하지 않는다.
2. 공수 계산 불일치가 저장 전에 막힌다.
3. 수동 수정은 명시적 예외로만 남는다.
4. 업로드, 다운로드, 반영이 같은 검증 결과를 공유한다.
5. 경계 시간과 예외는 경고로 분리되어 사용자가 확인할 수 있다.
